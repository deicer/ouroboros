# Мониторинг автономности (2026-02-25)

Источник: `data/logs/events.jsonl`, `data/logs/thinking_trace.jsonl`, `data/logs/chat.jsonl`, `data/logs/supervisor.jsonl`  
Цель: выявить, что мешает боту действовать самостоятельно и по делу.

## Ключевые наблюдения

1. Повтор шаблонного ответа на разные вопросы пользователя.
- В 16:38 UTC вопрос: `что делаешь?`
- В 16:39 UTC новый вопрос: `в чем проблема? почему долго?`
- Ответ почти тот же самый "фактологичный апдейт" (включая старый таймстамп 16:38 UTC).
- Признак: `tool_calls=0`, длинный общий статус вместо адресного ответа.

2. Зацикливание в фоне на чтении одних и тех же файлов.
- Фоновое сознание часто крутит 5 раундов с `repo_list/repo_read/drive_read`.
- Много циклов завершается пустым `thought_preview`.
- В прошлых задачах уже был stop по repeated-tool guard (`task_loop_guard_stop`).

3. Высокий контекст для простых запросов.
- Для коротких сообщений пользователя однораундовые ответы съедают ~29k prompt tokens.
- Это снижает релевантность и усиливает "инерцию" старого контекста.

4. Историческая проблема с двойным путем `data/data/...` все еще видна в файловом дереве.
- В `thinking_trace` есть `repo_list dir=data` с `data/data/`.
- Это может путать агента при диагностике состояния.

5. `tools_stats.jsonl` имеет владельца `root` (не `ouroboros`).
- Риск: недозапись статистики инструментов после рестартов/смены пользователя процесса.
- Потеря метрик ухудшает самообучение.

## Что именно мешает автономности

1. Нет "проверки релевантности ответа" перед отправкой пользователю.
- Бот может отправить красивый, но неответный и устаревший статус.

2. Нет жесткого лимита на "бесполезные фоновые раунды".
- Фоновое сознание иногда занимается обзором кода без конкретного следующего действия.

3. Недостаточно жесткое разделение "внутренний отчет" vs "ответ владельцу".
- В чат просачиваются блоки, похожие на внутренние отчеты.

4. Отсутствует автоматическая очистка legacy-артефактов пути.
- `data/data/` остается в видимом контексте и провоцирует ложные гипотезы.

## Рекомендации (для внедрения)

1. Ввести `response_relevance_guard` перед отправкой владельцу.
- Если ответ:
  - слишком похож на предыдущий (>90%),
  - и не содержит ключевых слов текущего вопроса,
  - и начинается с шаблона статуса (`"## Фактологичный апдейт"`, `"## Фоновый цикл"`),
  - то не отправлять сразу, а выполнить 1 re-ask к LLM:
    "Ответь прямо на последний вопрос пользователя в 2-4 предложениях, без статусного шаблона."

2. Для коротких direct-chat задач использовать компактный контекст-профиль.
- Для сообщений < 200 символов: урезать историю, убрать длинные отчеты, сократить trace до минимума.

3. Ограничить фоновое сознание по полезности.
- Max 1-2 tool calls за цикл без явной цели.
- Если нет конкретного action-item, завершать цикл одной краткой рефлексией.

4. Добавить startup-cleanup для legacy пути.
- Если найден `data/data/`, разово мигрировать/архивировать и скрыть из активного контекста.

5. Исправить права логов метрик.
- На старте проверять владельца `tools_stats.jsonl`, при необходимости чинить или писать в fallback-файл.

## Быстрые метрики контроля

1. `template_reply_rate`:
- доля ответов, начинающихся с "Фактологичный апдейт/Фоновый цикл" на direct questions.

2. `consciousness_empty_cycle_rate`:
- доля циклов, где `thought_preview` пуст.

3. `direct_chat_prompt_tokens_p50/p95`:
- токены на простых вопросах.

4. `repeated_guard_stops_per_day`:
- частота остановок по repeated tool batch/error.
